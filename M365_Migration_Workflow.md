# DPRP Cash Call Automation — Microsoft 365 Implementation Workflow

**Source system:** FastAPI + HTMX + PostgreSQL Cash Call Automation
**Target stack:** SharePoint Online Lists, Microsoft Forms, Power Apps (canvas), Power Automate, Power BI
**Document date:** 2026-06-12

This document maps the existing custom-built system onto the Microsoft 365 Power Platform, preserving the exact 6-stage approval chain, line-item deferral mechanics, urgent/standard branching, multi-currency line items, budget over-limit flagging, and full audit trail.

---

## 0. System Overview

The Cash Call lifecycle is a linear approval chain with three return paths and a per-line-item deferral capability:

```
Originator → HOD → Finance QC → CFO → CEO → Treasury (Paid)
                 ↘ return       ↘ return/query   ↘ defer line items
```

| Concept | Source value |
|---|---|
| Request types | standard, urgent |
| Currencies | USD, NGN, EUR, GBP, INR |
| Cost types | opex, capex |
| Frequencies | one_off, monthly, quarterly, annual |
| Line items per submission | 1–10 |
| Roles | originator, hod, finance_reviewer, cfo, ceo, treasury, it_admin |
| Budget enforcement | Warning only (flag, non-blocking) |

## 2. Entry Forms (Microsoft Forms + Power Apps)

### 2.1 Forms vs Power Apps split

Microsoft Forms **cannot** handle this entry process. It does not support: repeating sections (the 1–10 line-item gallery), cross-list lookups (exchange rate, budget), computed fields (USD equivalent), conditional multi-field branching beyond simple branching, or writing to two related lists in one transaction.

**Decision: use a Power Apps canvas app for all data entry.** Reserve Microsoft Forms only for lightweight ancillary intake — e.g. a "Request a new vendor code" or "Report an issue to IT Admin" form that triggers a separate flow. The core Cash Call submission is Power Apps only.

### 2.2 Power Apps screens

| Screen | Purpose |
|---|---|
| `scrHome` | Role-aware landing; tiles for New Submission / My Submissions / My Approvals |
| `scrHeader` | Department, Month/Year, Cost Type (OPEX/CAPEX), Justification, Request Type toggle |
| `scrUrgentDetails` | Visible only when RequestType = Urgent: urgency category, requested payment date, finance authoriser, urgency reason |
| `scrLineItems` | Gallery-based add/remove of up to 10 line items |
| `scrReview` | Read-only summary, total USD, budget over-limit banner, Submit button |
| `scrMyApprovals` | Stage-specific approval queue (deep-linked from email) |
| `scrSubmissionDetail` | Full record + audit timeline (read from AuditLog) |

### 2.3 Line-item dynamic add/remove (gallery pattern)

Use an in-memory collection bound to an editable gallery:

- Collection `colLineItems` holds draft rows. `+ Add Line Item` runs `Collect(colLineItems, {VendorName:"", Currency:"USD", ...})` — disabled (`DisplayMode.Disabled`) once `CountRows(colLineItems) >= 10`.
- Each gallery row uses `RemoveIf(colLineItems, ThisItem.GUID = ...)` on its delete icon.
- **USD equivalent** computed live per row: on Currency/Amount change, `LookUp(ExchangeRates, Title = Currency && EffectiveFrom <= Today() && IsBlank(ValidUntil) || ValidUntil >= Today()).RateToUSD`, then `EquivalentUSD = OriginalAmount / RateToUSD`. The rate is stamped into the row so it stays immutable after submit.
- On Submit, a single Power Automate flow (called via `PowerApps → Run`) receives the header JSON plus the line-item collection as a JSON array and writes parent + children, ensuring atomicity in one place.

### 2.4 Urgent vs standard conditional fields

- A toggle `tglRequestType` drives `Visible` on `scrUrgentDetails`.
- Validation rule on Submit button `DisplayMode`: if Urgent, require UrgencyCategory, RequestedPaymentDate, FinanceAuthoriser, UrgencyReason all non-blank; if Standard, those four are skipped/cleared.

## 3. Approval Workflow (Power Automate)

### 3.1 Structure: one main flow + child flows

- **Main flow `Flow_SubmitCashCall`** — trigger: *PowerApps (V2)*. Generates `SC-YYYY-NNNN`, looks up exchange rates, computes total USD, runs the budget check, creates the Submissions item and the LineItems items, writes the `submission_created` AuditLog entry, sets Status = `PendingHOD`, and starts the stage router.
- **Child flows** (called via *Run a Child Flow*), one per stage, to keep each approval block isolated and reusable: `Child_HODApproval`, `Child_FinanceQC`, `Child_CFOApproval`, `Child_CEOApproval`, `Child_TreasuryPayment`. Each child flow takes the Submission ID, runs its approval, writes its AuditLog row, sets the next Status, and returns an outcome to the main flow which then routes to the next child or terminates.

This avoids a single 200-action monolith and lets each stage be tested independently.

### 3.2 Triggers per stage

| Stage | Trigger pattern |
|---|---|
| Submit | PowerApps (V2) — explicit call from canvas app |
| HOD / Finance / CFO / CEO / Treasury | Invoked as child flow by the main orchestrator; **and/or** SharePoint *When an item is created or modified* on Submissions, gated by a Status filter (`triggerBody Status eq 'PendingHOD'` etc.) for resilience if a child flow is re-entered |

Recommended primary design: orchestrated child flows (deterministic). The SharePoint modified-trigger flows are the fallback/idempotent path.

### 3.3 Approval actions

Each stage uses the **Approvals** connector → *Start and wait for an approval*.

| Stage | Approval type | Response options | Assigned to |
|---|---|---|---|
| HOD | Custom Responses – Wait for one | Approve, Return for Revision, Decline | Department HOD (from Users list) |
| Finance QC | Custom Responses – Wait for one | Approve, Return, Raise Query | finance_reviewer |
| CFO | Custom Responses – Wait for one | Approve All, Defer Line Items, Decline | cfo |
| CEO | Custom Responses – Wait for one | Final Approve, Decline | ceo |
| Treasury | Custom Responses – Wait for one | Mark Paid, + 8 other payment statuses | treasury |

After each, a **Switch** on `outputs('Start_and_wait_for_an_approval')?['responses'][0]['responseSummary']` branches the workflow.

### 3.4 Conditional branches

**HOD (`Child_HODApproval`)**
- Approve → Status `PendingFinanceQC`, HODDecision Approved → invoke Finance QC.
- Return for Revision → Status `ReturnedForRevision` (HOD path), HODDecision Returned → notify originator, **end chain** (re-entry on resubmit, see 3.6).
- Decline → Status `DeclinedByHOD` → notify, terminate (terminal).

**Finance QC (`Child_FinanceQC`)**
- Approve → Status `PendingCFO` → invoke CFO.
- Return → Status `ReturnedForRevision` → back to originator.
- Raise Query → Status `QCQueryRaised` → back to originator; on resubmit returns to `PendingFinanceQC`.

**CFO (`Child_CFOApproval`)**
- Approve All → Status `PendingCEO` → invoke CEO.
- Decline → Status `DeclinedByCFO` → terminate (terminal).
- Defer Line Items → see 3.5.

**CEO (`Child_CEOApproval`)**
- Final Approve → for each non-deferred LineItem set `ApprovedUSD = EquivalentUSD`; Status `PendingTreasuryPayment` → invoke Treasury.
- Decline → Status `DeclinedByCEO` → terminate (terminal).

**Treasury (`Child_TreasuryPayment`)**
- Mark Paid → Status `Paid`, TreasuryPaymentStatus Paid; increment Budgets `PaidMTD`.
- Any other status → stays `PendingTreasuryPayment`, record TreasuryPaymentStatus.

### 3.5 CFO partial defer (line-item level)

This is the most custom piece. Implement inside `Child_CFOApproval`:

1. After the CFO chooses **Defer Line Items**, present a second approval/adaptive card (or a Power Apps mini-screen launched via deep link) listing each LineItem with a "defer to month" picker, so the CFO selects *which* items and the target month (1–12).
2. *Get items* (LineItems where SubmissionRef = ID). *Apply to each*: if selected, *Update item* `CFODeferred = Yes`, `CFODeferToMonth = chosenMonth`. For each deferred item, increment Budgets `DeferredApproved` for (Department, target month, year) — pre-reserving the amount, matching the source system.
3. Branch on count:
   - **All items deferred** → Status `DeferredByCFO`, CFODecision Deferred, CFOPostDeferral Yes → terminate chain (terminal, matches source).
   - **Some items deferred** → Status `PendingCEO`; remaining (non-deferred) items proceed to CEO. CEO approval then stamps `ApprovedUSD` only on the non-deferred items.
4. Write AuditLog `cfo_deferred` with notes `"CFO {name} deferred {x} of {total} item(s) to month {m}. Reason: {reason}"`.

> Note: Power Automate Approvals cannot natively render a per-row checkbox grid. The robust pattern is an **Adaptive Card** posted to Teams (Input.Toggle per line item + Input.ChoiceSet for month) or a dedicated Power Apps defer screen. The flow waits on that response, not on a vanilla approval.

### 3.6 HOD "Return for Revision" (reopen to originator)

- On Return, set Status `ReturnedForRevision` and notify the originator (Office 365 Outlook *Send an email (V2)* with deep link `https://apps.powerapps.com/...&recordId={id}`).
- The originator reopens the record in `scrHeader`/`scrLineItems` (editable because Status is a revision state). On re-submit, a *When item modified* flow filtered to `ReturnedForRevision`/`QCQueryRaised` resets Status to the correct re-entry point (`PendingHOD` after HOD return; `PendingFinanceQC` after Finance return/query) and writes a `resubmitted` AuditLog row, re-invoking the chain from that stage.

### 3.7 Email notifications

Connector: **Office 365 Outlook → Send an email (V2)** (or Teams *Post message*). Triggered on: submission created (to HOD), each stage advance (to next approver), each return/query/decline (to originator), CFO defer (to originator + CEO), CEO approve (to Treasury), and Paid (to originator + Finance). Every email carries the Submission ID, total USD, urgency banner if urgent, and a deep link.

### 3.8 Budget over-limit (in `Flow_SubmitCashCall`)

Mirror the source `check_budget`: *Get items* Budgets for (Department, Month, Year). If none or `MonthlyAllocationUSD = 0` → flag false. Else `remaining = MonthlyAllocationUSD - ApprovedMTD - DeferredApproved`; `overage = TotalRequestedUSD - remaining`; set `BudgetOverLimitFlag = overage > 0`, `BudgetOverageUSD = max(overage,0)`. Non-blocking — submission proceeds either way.

## 4. Reporting & KPIs (Power BI)

### 4.1 Dataset connections

- Connector: **SharePoint Online List** (`SharePoint.Tables` / the list connector), one query per list: Submissions, LineItems, Budgets, ExchangeRates, AuditLog, Users.
- Model relationships: Submissions[Title] 1—* LineItems[SubmissionRef]; Submissions 1—* AuditLog[SubmissionRef]; Budgets joined on a composite `Department|Month|Year` key built in Power Query.
- Add a Date dimension table for time intelligence (cycle time, MTD/YTD).

### 4.2 Key visuals

| KPI | Visual type | Definition |
|---|---|---|
| Spend by department | Clustered bar chart | SUM(LineItems[EquivalentUSD]) by Submissions[Department] |
| OPEX vs CAPEX split | Donut chart | by Submissions[CostType] |
| Submission status pipeline | Funnel chart (or stacked column) | COUNT of Submissions by Status in chain order |
| Budget vs actual | Clustered column + line, or KPI card | Budgets[MonthlyAllocationUSD] vs ApprovedMTD/PaidMTD |
| Urgent vs standard ratio | 100% stacked bar / gauge | COUNT by RequestType |
| Approval cycle time | Line chart / matrix | DATEDIFF(Created, CEODecidedAt) avg by month; from AuditLog timestamps |
| Currency breakdown | Treemap or stacked bar | SUM(OriginalAmount) by Currency |
| Budget over-limit submissions | Card + table | COUNT where BudgetOverLimitFlag = Yes |

Measures (DAX) such as `Avg Cycle Days = AVERAGEX(Submissions, DATEDIFF(Submissions[Created], Submissions[CEODecidedAt], DAY))`.

### 4.3 Refresh

- **Scheduled refresh 8×/day** (Power BI Pro on SharePoint sources allows up to 8) — e.g. hourly during business hours. SharePoint lists do not support DirectQuery, so near-real-time is not achievable; Import + scheduled refresh is the model.
- For approval-desk dashboards needing live status, surface those directly in Power Apps off the live lists rather than Power BI.

## 5. Security & Access Control

### 5.1 SharePoint permission groups → roles

| Role | SharePoint group | List permissions |
|---|---|---|
| originator | CashCall-Originators | Submissions/LineItems: Contribute (own items via item-level perms); AuditLog: Add-only |
| hod | CashCall-HODs | Submissions: Edit (approval fields); read all in own dept |
| finance_reviewer | CashCall-Finance | Submissions: Edit; ExchangeRates: Edit |
| cfo | CashCall-CFO | Submissions + LineItems: Edit |
| ceo | CashCall-CEO | Submissions + LineItems: Edit |
| treasury | CashCall-Treasury | Submissions: Edit treasury fields; Budgets: Edit |
| it_admin | CashCall-Admins | Full Control; Users + ExchangeRates + Budgets management |

AuditLog: everyone **Add items** only (no Edit/Delete) to preserve immutability — enforced via list-level permission break.

### 5.2 Power Apps role-based screen visibility

- On app start, `Set(varRole, LookUp(Users, UserAccount.Email = User().Email).Role)`.
- `scrMyApprovals` filters the queue by role↔status (HOD sees `PendingHOD` for their dept, CFO sees `PendingCFO`, etc.).
- Decision buttons use `Visible = varRole = "CFO"` style gating; `scrHeader` editable only when originator owns the record and Status is a draft/revision state.

### 5.3 Power BI row-level security (RLS)

- Define RLS roles: `Dept_HOD` filters `Submissions[Department] = LOOKUPVALUE(Users[Department], Users[Email], USERPRINCIPALNAME())`; `Originator` filters to own submissions.
- CFO/CEO/Finance/Treasury/Admin roles see all rows (no filter). Map AAD groups to these RLS roles in the Power BI Service dataset security settings.

## 6. Limitations vs the Custom-Built FastAPI System

| Capability | FastAPI system | M365 stack |
|---|---|---|
| Parent/child atomicity (Submission + 10 LineItems) | True DB transaction | No transaction — flow writes sequentially; partial-failure cleanup must be coded manually |
| Composite uniqueness (budget key, SC-IDs) | DB constraints | No native enforcement; must validate in app/flow (race conditions possible) |
| Immutable exchange-rate stamping | Numeric(18,8) column, enforced | Stamped as value, but no DB-level immutability guarantee |
| Per-line-item CFO defer UI | Native HTMX UI | Requires Adaptive Card or extra Power Apps screen — Approvals can't render a row grid |
| Audit trail immutability | Append-only table + app logic | Add-only SharePoint perms approximate it; admins can still alter |
| Currency precision | Decimal(18,2)/(18,8) exact | SharePoint Number/Currency uses floating point — rounding risk on FX math |
| Reporting latency | Live queries | Power BI Import with ≤8 refreshes/day; no DirectQuery on lists |
| Complex business rules | Arbitrary Python | Power Fx + flow expressions — verbose, harder to unit test |
| List throttling / scale | PostgreSQL scales | SharePoint 5,000-item view threshold; large LineItems/AuditLog need indexed columns and care |
| Branch/version control of logic | Git | Power Platform solutions + environments; weaker diffing |
| Performance under load | Tunable | Power Automate throttling, daily action limits per license |

**What the M365 stack does well here:** no servers to run; native AAD identity and email; built-in Approvals with Teams/Outlook actionable cards; Power BI governance and RLS; low-code maintenance by business analysts; tight Microsoft 365 integration.

**What it cannot match cleanly:** transactional integrity across the parent/child write, exact decimal FX arithmetic, the bespoke per-line-item defer interaction, true real-time dashboards, and rigorous automated testing of the workflow logic. These should be acknowledged as accepted trade-offs or mitigated (e.g. Dataverse instead of SharePoint Lists for transactions and stronger relationships, if licensing permits).

> **Recommendation:** If transactional integrity, relational constraints, and per-row currency precision are hard requirements, migrate the data layer from SharePoint Lists to **Microsoft Dataverse** while keeping the same Power Apps / Power Automate / Power BI layers. Dataverse closes most of the gaps in the table above at higher licensing cost.
